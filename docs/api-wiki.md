## Определения
* Чанк &mdash; массив байт, часть файла. Обычно в разы меньше самого файла.

## Endpoints
### Статусы авторизации
Почти каждый ендпоинт требует авторизованного пользователя. Для этого используется `JWT` токен, передаваемый в заголовке `Authorization` запроса.

У каждого ендпоинта, требующего авторизации, есть общие статусы и ошибки:
* 400 (Bad request) &mdash; токен записан в неправильной форме
* 400 (Bad request) &mdash; токен имеет неверную сигнатуру
* 401 (Unauthorized) &mdash; поле `Authorization` отсутствует, либо пустое
* 401 (Unauthorized) &mdash; `jwt` токен просрочен

### Регистрация
✳️ `POST /api/v1/users/register`

Создаёт учётную запись пользователя на сервере.

#### Тело запроса
Тело представляет собой объект типа `application/json`. Пример:

``` json
{
	"username": "anton",
	"password": "123",
	"key": "59addea5aacf5ccb832921cca9f2e22d04a47670202a3da9557bda9fc2578bf4"
}
``` 

* `username`: имя пользователя, уникальное в системе
* `password`: пароль пользователя
* `key`: секретный ключ получаемый при настройке сервера

#### Тело ответа
Представляет собой описание ошибки, если она есть, и ничего в случае успешной регистрации.

#### Статусы
* 100 (Continue) &mdash; пользователь уже зарегистрирован
* 200 (OK) &mdash; пользователь успешно зарегистрирован
* 400 (Bad request) &mdash; пустое тело запроса
* 400 (Bad request) &mdash; ошибка в синтаксисе `JSON`
* 400 (Bad request) &mdash; пустое поле с именем пользователя
* 403 (Forbidden) &mdash; введённый секретный ключ не существует
* 404 (Not found) &mdash; сервис аутентификации недоступен
* 500 (Internal error); &mdash; внутренняя ошибка сервиса

### Авторизация
✳️ `POST /api/v1/users/login`

Возвращает `JWT` токен пользователя. Токен действует 24 часа, далее требуется повторная авторизация.

#### Тело запроса
Тело представляет собой объект типа `application/json`. Пример:

``` json
{
    "username": "Ivan", 
    "password": "pass123"
}
``` 

* `username`: имя пользователя
* `password`: пароль пользователя

#### Тело ответа
Представляет собой описание ошибки, если она есть, а также текст с `JWT` токеном, при успешной авторизации.

#### Статусы
* 200 (Ok) &mdash; авторизация прошла успешно
* 400 (Bad request) &mdash; пустое тело запроса
* 400 (Bad request) &mdash; ошибка в синтаксисе `JSON`
* 400 (Bad request) &mdash; пустое поле с именем пользователя
* 400 (Bad request) &mdash; пользователь не зарегистрирован
* 400 (Bad request) &mdash; введён не верный пароль
* 404 (Not found) &mdash; сервис аутентификации недоступен
* 500 (Internal error); &mdash; внутренняя ошибка сервиса

### Создание файлового соединения
✳️ `POST /api/v1/files/connect?mode`

"Пропуск" к работе с файлами на сервере. Создаёт инструкцию для сервера, что конкретный файл будет использоваться (создание, изменение и т.д.). Без этой инструкции, работа с файлами не допускается.

Также передаёт пользователю информацию о количестве чанков и их размере, необходимые для сохранения файла. 

Соединение существует 5 минут.

> [!IMPORTANT]
> При сохранении файла, пользователь сможет сохранить столько чанков, сколько указано в ответе, даже если соединение всё ещё будет существовать.

#### Параметры URL
##### mode 
Тип создаваемого соединения.

Существует два типа подключения:
* `RDWR` - подключение для создания, редактирования и чтения файлов. При использовании - обязательно указывать размер файла в теле запроса. При использовании данного мода для создания соединений к уже существующим файлам - новые сохраняемые чанки будут перезаписывать старые.

* `RDONLY` - подключение только для чтения фаилов. При использовании - указывать размер файла не обязательно.

#### Заголовки запроса
Запрос должен иметь заголовок `Authorization` с `JWT` токеном.

#### Тело запроса
Тело представляет собой объект типа `application/json` со структурой:

``` json
{
    "directory": "/",
    "filename": "diary.txt",
    "size": 1024
}
``` 

* Поля `directory` и `filename` являются обязательными. Поле `size` следует указывать только при типе подключения `RDWR`.
* Поле `directory` всегда должно начинаться и оканчиваться `/`
* В поле `size` размер файла указывается в байтах. 

#### Тело ответа
Тело ответа представляет собой текст ошибки, если она есть, или объект типа `application/json` со структурой:

``` json
{
    "UUID": "fbcb8d8b-b53e-4e01-92d0-d1dc3e0dffa8",
    "chunkSize": 1024,
    "chunksCount": 1
}
```

* Поле `UUID` используется для остальных запросов файловому сервису.
* `chunkSize` и `chunksCount` &mdash; информация как передавать файлы для сохранения.

#### Статусы
* [Статусы авторизации](#cтатусы-авторизации)
* 200 (Ok) &mdash; соединение создано
* 400 (Bad request) &mdash; тело запроса пустое
* 400 (Bad request) &mdash; пустое имя файла
* 400 (Bad request) &mdash; указан неизвестный тип соединения
* 400 (Bad request) &mdash; не указан каталог файла
* 400 (Bad request) &mdash; указанный каталог записан в неправильно форме
* 400 (Bad request) &mdash; указанный каталог не найден
* 400 (Bad request) &mdash; не указан размер сохраняемого файла
* 413 (Request entity too large) &mdash; размер сохраняемого файла, больше свободного места на сервере
* 500 (Internal error); &mdash; внутренняя ошибка сервиса
* 503 (Service unavailable) &mdash; файловый сервис недоступен

### Сохранение файлов
✳️ `POST /api/v1/files/save?connID`

Используется для сохранения файла по частям (далее `чанк`). Сохраняет чанк на сервере. Чтобы сохранить полный файл, необходимо отправить запрос столько раз, сколько указано в поле `chunksCount` созданного [соединения](#cоздание-файлового-соединения)

#### Параметры URL
* `connID` &mdash; UUID [соединения](#cоздание-файлового-соединения)

#### Тело запроса
Тело представляет собой объект типа `application/json` со структурой:

``` json
{
    "chunk": [104, 101, 108, 108, 111],
    "offset": 0
}
```

* Поле `chunk` &mdash; массив байт, который будет сохранён на сервере. Размер чанка должен быть не больше указанного при создании соединения.

* `offset` &mdash; отступ сохранения. В примере указано значение 0, значит сервер отступит от начала файла 0 байт и сохранит чанк. Предположим, значение `offset` равно 5, значит сервер отступит 5 байт от начала файла и сохранит чанк.

> [!CAUTION]
> Сервер перезапишет уже записанные байты файла, на данные из запроса, при неправильном `offset`.

#### Тело ответа
Представляет собой описание ошибки, если она есть, и ничего в случае успешном сохранении чанка.

#### Статусы
* [Статусы авторизации](#cтатусы-авторизации)
* 200 (Ok) &mdash; чанк успешно сохранён
* 400 (Bad request) &mdash; uuid записан в неправильной форме
* 400 (Bad request) &mdash; пустое тело запроса
* 400 (Bad request) &mdash; размер чанка больше, указанного в соединении
* 400 (Bad request) &mdash; соединения не существует
* 403 (Bad request) &mdash; попытка сохранить количество чанков, которое больше указанного при создании соединения.
* 403 (Bad request) &mdash; попытка сохранения при типе соединения `RDONLY`
* 500 (Internal error) &mdash; внутренняя ошибка сервиса
* 503 (Service unavailable) &mdash; файловый сервис не доступен

### Получение файла
✳️ `GET /api/v1/files/get?connID&chunkID`

Используется для получения файла с сервера. Запрос возвращает файл по чанкам. Чтобы получить полный файл, необходимо отправить столько запросов, сколько записано в поле `chunksCount` при [создании соединения](#cоздание-файлового-соединения).

#### Параметры URL
* `connID` &mdash; UUID [соединения](#cоздание-файлового-соединения)
* `chunkID` &mdash; параметр говорит серверу, какой по счёту чанк файла вернуть пользователю. Соответственно, этот параметр должен быть положительным и целочисленным (int).

#### Тело ответа
Представляет собой описание ошибки, если она есть, или `octet-stream` контент &mdash; массив байт (чанк файла).

#### Статусы
* [Статусы авторизации](#cтатусы-авторизации)
* 200 (Ok) &mdash; чанк получен успешно
* 400 (Bad request) &mdash; uuid записан в неправильной форме
* 400 (Bad request) &mdash; пустое тело запроса
* 400 (Bad request) &mdash; соединения не существует
* 400 (Bad request) &mdash; ID соединения имеет не UUID форму
* 500 (Internal error) &mdash; внутренняя ошибка сервиса
* 503 (Service unavailable) &mdash; файловый сервис не доступен

### Получение контрольной суммы
✳️ `GET /api/v1/files/sum?connID&chunkID`

Используется проверки целлостности сохранённого файла. Возвращает контрольную сумму чанка (не файла!), поэтому, чтобы проверить целостность файла, необходимо сделать столько запросов, сколько указано в поле `chunksCount` при [создании соединения](#cоздание-файлового-соединения)

Контрольная сумма считается по алгоритму SHA-256.

#### Параметры URL
* `connID` &mdash; UUID [соединения](#cоздание-файлового-соединения)
* `chunkID` &mdash; параметр говорит серверу, контрольную сумму какого по счёту чанка нужно вернуть. Соответственно, этот параметр должен быть положительным и целочисленным (int).

#### Тело ответа
Ответ представляет собой текст с ошибкой, если есть, либо `octet-stream` контент &mdash; массив байт (контрольную сумму).

#### Статусы
* [Статусы авторизации](#cтатусы-авторизации)
* 200 (Ok) &mdash; контрольная сумма успешно создана
* 400 (Bad request) &mdash; uuid записан в неправильной форме
* 400 (Bad request) &mdash; пустое тело запроса
* 404 (Bad request) &mdash; соединения не существует
* 400 (Bad request) &mdash; ID соединения имеет не UUID форму
* 500 (Internal error) &mdash; внутренняя ошибка сервиса
* 503 (Service unavailable) &mdash; файловый сервис не доступен

### Получение списка файлов директории
✳️ `GET /api/v1/files?dir`

Используется для получения списка файлов и папок в указанной директории.

#### Параметры URL
* `dir` &mdash; каталог из которого нужно получить список файлов. Параметр должен начинаться и заканчиваться `/`.

#### Тело ответа
В качестве ответа возвращается текст ошибки (при её наличии), либо `JSON` структура при успешном выполнении запроса.

``` json
[
  {
    "name": "photos",
    "isDir": true,
    "modTime": 1768085187
  },
  {
    "name": "my diary.txt",
    "size": 1,
    "modTime": 1767653758
  },
  {
    "name": "pentagon secrets.docx",
    "size": 1855172345498,
    "modTime": 1768085187
  },
  {
    "name": "cookie recipe.txt",
    "size": 150,
    "modTime": 1765733471
  }
]
```

`JSON` объект имеет массив, c структурой:
* `name` &mdash; имя файла
* `isDir` &mdash; является ли файл - каталогом. Указывается только при положительном значении.
* `size` &mdash; размер файла в байтах (указывается только не каталогов)
* `modTime` &mdash; время создания файла
